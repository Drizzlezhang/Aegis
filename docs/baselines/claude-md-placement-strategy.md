# 目录级 CLAUDE.md 落位策略（Phase 1）

> 文档定位：本文件是“规则落位策略”，用于指导后续将现有规则分层沉淀到不同目录级 `CLAUDE.md`。  
> 非目标：本文件**不直接生成**根目录、`web/`、`src/` 或二级目录的 `CLAUDE.md` 具体正文。

## 1. 目标与边界

### 1.1 目标
- 建立一套可执行的规则分层机制，明确“哪类规则放在哪一层”。
- 降低根 `CLAUDE.md` 体积与重复度，减少跨任务上下文噪音。
- 避免同一规则在多层重复拷贝导致冲突与漂移。

### 1.2 边界
- 本阶段只产出落位策略，不变更既有规则文档内容。
- 本阶段不新增目录级 `CLAUDE.md` 实体文件（后续 Phase 2 按策略落地）。
- 规则落位仅覆盖当前仓库内治理文档体系，不扩展到外部仓库。

---

## 2. 分层模型与职责边界

采用四层治理模型：

- **L0（仓库根）**：全局执行契约与跨目录约束
- **L1（域目录）**：`web/`、`src/` 的领域规则
- **L2（二级稳定子目录）**：例如 `src/agents/`、`src/skills/`；若前端后续稳定收敛为 feature 组织，再评估 `web/features/` 等目录
- **L3（实现目录）**：普通业务目录，不默认放 `CLAUDE.md`

核心原则：
- 上层只定义跨域共识与硬约束，不写具体实现细节。
- 下层只补充本域特有规则，不重复上层通则。
- 低层规则不得与高层冲突；冲突时以上层为准并触发修订流程。

---

## 3. 根 `CLAUDE.md` 应保留内容（L0）

根文件只保留“全局必须一致”的规则，建议包含以下四类：

1. **全局工作流**
   - 需求澄清 → 计划 → 执行 → 验证 → 收尾 的统一流程。
   - 非 trivial 任务先计划、执行后给验证和收尾建议。

2. **跨目录依赖与边界总则**
   - `web/` 与 `src/` 的协作关系、变更影响识别要求。
   - 跨层改动时的拆分原则（按模块边界、接口先行、读写分离）。

3. **风险控制总则**
   - 全局高风险变更（全局配置、认证、CI/CD、共享接口）需先提醒再执行。
   - 覆盖未提交改动、潜在破坏性操作的防护要求。

4. **验证与收尾通则**
   - 测试、lint/typecheck/build、手动回归路径的最小检查清单。
   - 临时代码清理、无关变更检查、commit/PR 描述完整性要求。

不应放在根文件的内容：
- 前端目录结构细则、页面迁移细节。
- 后端具体模块依赖矩阵、子模块实现约束。
- 仅在单一域生效的 i18n/theme 或 agent/skill 细则。

---

## 4. `web/CLAUDE.md` 应沉淀内容（L1-Web）

`web/` 层聚焦前端域内约束，建议沉淀：

> 说明：本节中的 `feature / shared / foundation` 是**职责分组模型**，用于指导规则归类；在 Phase 2 物理目录方案未最终确定前，不等同于当前仓库已存在或已承诺落地的目录结构。

1. **目录职责边界**
   - `feature/`：业务能力与场景逻辑。
   - `shared/`：跨 feature 复用组件与能力。
   - `foundation/`：基础设施（主题、国际化、网络、状态基座等）。

2. **i18n / theme / api 边界**
   - i18n 文案来源与禁止硬编码规则。
   - 主题 token / 颜色语义来源与禁止组件内私有语义漂移。
   - API 调用入口与数据映射边界（页面层不直接拼接后端协议细节）。

3. **页面迁移约束**
   - 迁移顺序：先稳定共享层，再迁移页面场景。
   - 迁移期间兼容策略：允许适配层，不允许长期双轨重复实现。
   - 迁移完成判据：旧入口摘除、依赖收敛、关键路径回归通过。

4. **UI 不变量**
   - 双语兼容基线（`zh-CN`/`en` 同步）。
   - 行情颜色语义统一（上涨红、下跌绿）及共享映射优先。
   - 加载态/空态/错误态完整性。

不应在 `web/CLAUDE.md` 重复的内容：
- 根层通用工作流、全局风险流程。
- 后端 `src/` 的层次依赖细节。

---

## 5. `src/CLAUDE.md` 应沉淀内容（L1-Src）

`src/` 层聚焦后端域内工程边界，建议沉淀：

1. **后端层次边界**
   - `api`（接口暴露）/ `agents`（编排与决策）/ `skills`（能力插件）/ `llm`（模型路由）职责分离。
   - 禁止跨层绕行调用（例如业务层绕过规范入口直接触底层实现）。

2. **依赖方向（必须单向）**
   - 明确 `api/agents/skills/llm` 允许与禁止依赖关系。
   - 避免循环依赖，跨层交互优先通过契约接口。

3. **双轨治理规则**
   - 允许“存量轨 + 新架构轨”并存的条件、时限与退出标准。
   - 每次新增能力必须声明落在哪条轨道，禁止隐式扩散双轨。
   - 双轨收敛触发条件与回收流程（里程碑、覆盖率、回归质量）。

不应在 `src/CLAUDE.md` 重复的内容：
- 前端页面/主题/i18n 细节。
- 根层流程性通则。

---

## 6. 规则下钻策略：哪些不该继续下钻，何时允许下钻

### 6.1 不应继续下钻的规则类型
以下规则应停留在 L0 或 L1，不建议继续复制到 L2：
- 全局工作流与收尾流程。
- 全局风险控制和高危确认机制。
- 跨域协作协议（web ↔ src）。
- 已在 L1 明确且对该域全局生效的通用边界规则。

### 6.2 允许下钻到 L2 的触发条件
仅当同时满足以下条件，才允许新增二级目录 `CLAUDE.md`：
1. 目录职责长期稳定（至少跨 2 个迭代未发生结构重命名）。
2. 该目录存在**独有规则**，且无法被 L1 规则准确表达。
3. 该规则会被高频触发（例如每周多次改动该目录）。
4. 新规则不会复制上层条目，且可用“补充条款”形式表达。
5. 存在明确验收标准（可检查、可回归、可判定是否违背）。

## 7. 二级目录落位前置条件（示例）

### 7.1 `web/features/`（候选）
允许前置条件：
- 前端结构已稳定收敛为 feature 组织，且入口、依赖、复用关系清晰。
- 已形成统一的页面状态约束（loading/empty/error/skeleton）并可复用。
- 迁移策略已进入执行期，需要目录级约束防止回退到旧模式。

### 7.2 `src/agents/`
允许前置条件：
- Agent 协作边界已清晰（编排、输入输出契约、错误传播路径）。
- 多 Agent 变更频繁，且存在反复出现的越层调用问题。
- 已有可执行的验证清单（集成测试/链路回归）支撑规则落地。

### 7.3 `src/skills/`
允许前置条件：
- Skill 注册、元数据、生命周期边界已稳定。
- 新增 Skill 模板与校验项固定，具备可自动检查项。
- 目录内规则可显著减少“接口不一致/注册不完整”类问题。

---

## 8. 避免规则重复、冲突、漂移的机制

1. **单一真源（SSOT）**
   - 每条规则只在一个层级定义“主版本”；其他层仅引用，不复制粘贴。

2. **规则 ID 与引用机制**
   - 为关键规则分配 ID（如 `ROOT-RISK-01`、`WEB-I18N-02`、`SRC-DEP-03`）。
   - 下层文档通过“引用 ID + 本层补充”方式组织。

3. **冲突判定顺序**
   - L0 > L1 > L2。
   - 一旦出现语义冲突，低层规则立即标记废止并回滚到上层定义。

4. **变更门禁**
   - 规则改动 PR 需包含：影响层级、替换关系、迁移计划、验证方式。
   - 若涉及上层规则变更，必须同步检查所有引用层。

5. **定期对账**
   - 按迭代执行“规则对账”：去重、失效条款清理、引用有效性检查。

---

## 9. 与 baseline / 边界规则文档的关系

本文件与既有文档关系如下：

- `docs/baselines/current-architecture-baseline.md`：描述“当前架构现状（是什么）”。
- `docs/baselines/frontend-logic-boundary-rules.md`：描述“前端边界规则（前端应如何做）”。
- `docs/baselines/backend-logic-boundary-rules.md`：描述“后端边界规则（后端应如何做）”。
- **本文件**：描述“这些规则未来应放在哪一层（放在哪里）”。

结论：
- `docs/baselines/current-architecture-baseline.md` 是“现状事实源”。
- `docs/baselines/frontend-logic-boundary-rules.md` 与 `docs/baselines/backend-logic-boundary-rules.md` 是“边界规则源”。
- 本文件是“规则落位与演进路线图”。
- 后续新增目录级 `CLAUDE.md` 时，应以本策略进行拆分与引用，而不是重新发明规则。

---

## 10. Phase 2 执行建议（供后续落地）

1. 先冻结 Phase 1 的 baseline / boundary rule 文档版本，避免规则源在拆分过程中继续漂移。
2. 再做规则清单化：为根/前端/后端规则打 ID。
3. 然后做 L0/L1 初次落位：仅迁移高确定性规则。
4. 最后评估 L2：仅在触发条件满足时新增二级目录 `CLAUDE.md`。
5. 每轮落位后执行一次冲突与重复审计，确保文档体系收敛。
