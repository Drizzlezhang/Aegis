# Aegis-Trader 项目契约

## 项目定位
专注美股正股及期权（LEAPS, Bull Spreads, Covered Call）的左侧抄底策略的 Multi-Agent 量化交易系统。受 TauricResearch/TradingAgents 启发，但聚焦于**抄底+期权策略**这一垂直场景，引入机构级量化算法和动态 Skill 加载机制。

## 核心标的
- **ETF**: QQQ, SPY
- **科技股**: NVDA, MSFT, AAPL, PLTR, NFLX, INTC, TSM, TSLA
- **防御股**: KO

## 策略方向
- **左侧抄底**: 在支撑位附近建立仓位
- **期权策略**: 10个月+ LEAPS Call, Bull Spreads, Covered Calls
- **多维度支撑计算**: Volume Profile, GEX Wall, 用户先验点位
- **机构级估值**: PE-Band, 多因子模型, 机构共识分析

## 架构核心
- **Multi-Agent 系统**: Data-Harvester, Institutional-Analyst, Strategy-Execution, Aegis-Memory
- **Skill 动态加载**: 参考 Hermes Agent skills/plugins 架构，数据源和算法模块化
- **交易记忆系统**: 参考 Hermes memory plugin 体系，SQLite + 向量存储
- **模型路由**: deepseek-v3.2 (统一开发模型), 所有编码/规划/审查任务均使用 deepseek3.2

### 架构设计原则（上下文窗口友好）
为从根本上减少单个任务涉及的代码量，避免频繁 `/compact`，架构层面必须遵循以下原则：

#### 1. 文件粒度控制
- **每个源码文件不超过 300 行**（测试文件不超过 500 行）
- **每个函数不超过 50 行**，复杂逻辑拆分为多个小函数
- **每个类不超过 15 个方法**，过大则拆分为多个协作类
- 超过限制时，优先垂直拆分（按功能分层）而非水平拆分（同一层多个文件）

#### 2. 模块独立性（高内聚低耦合）
- **通过接口交互，而非直接依赖实现**：调用其他模块时，只依赖其公开接口，不依赖内部细节
- **单向依赖**：避免循环依赖，依赖关系图应为有向无环图（DAG）
- **黑盒原则**：修改模块 A 时，不需要了解模块 B 的内部实现，只需知道其接口契约

#### 3. 接口契约化
- 模块间交互通过明确的接口契约定义（函数签名、返回值类型、异常约定）
- 接口定义在独立文件中（如 `src/agents/base.py` 定义 `BaseAgent`，各 Agent 实现该接口）
- 修改接口时必须同时更新所有实现者，此类改动应作为独立任务

#### 4. 上下文隔离策略
- **垂直分层**：Data → Analysis → Strategy → Memory，每层只与相邻层交互
- **水平隔离**：各 Agent 之间不直接调用，通过 Orchestrator 协调
- **Skill 自治**：每个 Skill 独立完成一项功能，通过统一接口注册到 Registry

### 任务拆分原则
当用户提出一个可能涉及多个文件/模块的需求时，Claude **必须**主动按以下规则拆分：

1. **按模块边界拆分**：每个子任务只涉及一个模块或一个层次
2. **按读写分离拆分**：先完成所有"读"（分析现状），再开始"写"（修改代码）
3. **接口先行**：如需新增接口，先定义接口，再分别实现调用方和被调用方
4. **验证独立**：每个子任务完成后可独立验证，不依赖后续任务

## 模型路由策略

**本项目统一使用 `deepseek3.2` 作为唯一开发模型**。所有环节（Plan、Code、Review、Test）均使用 deepseek3.2，不再在不同模型间切换。

### 统一使用 deepseek3.2 的原因
- deepseek3.2 具备强大的代码生成、推理和规划能力，足以覆盖所有开发环节
- 避免多模型切换带来的上下文丢失和状态不一致
- 简化开发流程，减少模型切换打断

### deepseek3.2 上下文窗口管理（关键约束）
deepseek3.2 上下文窗口约为 **64k tokens**。**优先通过架构设计（文件粒度控制、模块独立性）减少上下文消耗，而非依赖频繁 `/compact`**。

#### 1. 文件读取控制
- **单次读取不超过 500 行**（除非文件确实更长且必须全读）
- **单次并行读取不超过 3 个文件**
- 优先使用 `smart_outline` 和 `smart_search` 快速定位代码结构，而非全量读取
- 需要读取大文件时，使用 `offset`/`limit` 分段读取

#### 2. 对话历史管理（备用策略）
- **架构设计是首选**：通过文件拆分和模块独立，让单个任务自然只涉及少量文件（参见上方"架构设计原则"）
- **`/compact` 是最后手段**：当对话轮次超过 15 轮或代码累积超过 5000 行时才建议 `/compact`
- **避免在单轮中堆叠过多工具调用**：每轮工具调用（Read/Edit/Bash）不超过 5 个
- **优先使用 Agent 工具分散上下文**：复杂多步骤任务拆分为子任务，由子 Agent 承担部分工作

#### 3. 代码修改策略
- **优先使用 Edit 而非 Write**：Edit 只传输 diff，显著减少上下文消耗
- **避免全文件重写**：除非文件确实需要完全重写，否则用 Edit 做精确替换
- **分批次提交修改**：单轮修改不超过 3 个文件，多文件改动分多轮完成

#### 4. 测试输出控制
- **运行测试时加 `--tb=short`** 或 `--tb=line`，避免长 traceback 撑满上下文
- **分段查看测试输出**：大型测试套件分批运行，而非一次性全部执行
- **使用 `| head -N` 截断输出**：Bash 命令输出过长时主动截断

## 风险控制
1. **AWS 内存约束**: 使用 SQLite 替代 PostgreSQL，避免内存密集型缓存
2. **数据源稳定性**: yfinance 为主，alpha_vantage 为备用，后续接长桥/富途
3. **GEX 精度**: yfinance 不提供精确 gamma 和 OI，初期用近似算法
4. **YouTube 抓取**: Phase 2 实现，注意反爬限制

### 上下文告警触发条件
`/compact` 是最后的兜底手段。**优先通过架构设计（小文件、模块独立）避免触及窗口限制**。仅在以下情况才提示 `/compact`：

1. 对话轮次 ≥ 15 轮（正常单任务不应超过此轮次）
2. 累积代码行数 ≥ 5000 行（包括读取和生成的代码）
3. 单次测试输出超过 800 行
4. 连续多轮每轮都修改了 3+ 文件且已持续 5 轮以上

**触发时的处理流程**：
1. 先检查是否可通过任务拆分解决（将剩余工作拆为独立子任务）
2. 如无法拆分，再建议 `/compact`
3. 提示语：”当前对话上下文已累积较多内容，建议执行 `/compact` 压缩历史。更优方案是将剩余工作拆分为独立子任务。”

## accept-edits 模式安全规范

用户使用 Claude Code 的 **accept-edits** 模式（Edit 工具自动执行，无需逐条确认）。为防止自动执行带来的风险，必须遵守以下安全规则。

### 自动执行前的强制说明
**任何 Edit 执行前，Claude 必须在回复中明确说明**：
1. **修改目标**：要改哪个文件的哪几行
2. **修改原因**：为什么要改，解决什么问题
3. **影响范围**：此修改会影响哪些其他文件/功能
4. **回退方式**：如何撤销此修改（如 `git checkout -- <file>`）

### 高危操作清单（执行前必须用户确认）
以下操作即使在 accept-edits 模式下，**也必须先获得用户明确确认**（不能自动执行）：

| 操作类型 | 说明 | 确认语 |
|----------|------|--------|
| 删除文件/目录 | `rm`、`rmdir`、删除大量代码 | "即将删除 [文件]，是否确认？" |
| 修改全局配置 | `pyproject.toml`、Dockerfile、CI/CD | "即将修改 [配置文件]，是否确认？" |
| 修改环境/密钥相关 | `.env`、`config.py` 中的 API key、token | "涉及敏感配置，请先确认" |
| 跨模块接口变更 | 修改 `BaseAgent`、`BaseSkill` 等接口 | "此修改会影响所有实现者，是否确认？" |
| 大规模重构 | 单次修改超过 5 个文件或超过 500 行 | "涉及大规模改动，建议拆分为子任务，是否继续？" |
| 数据库/存储操作 | 删除表、修改 schema、清理数据 | "涉及数据操作，是否确认？" |

### 禁止自动执行的行为
- **禁止自动运行可能破坏数据的命令**：如 `git reset --hard`、`git clean -f`、删除数据库
- **禁止自动修改未读过的文件**：必须先 Read 再 Edit
- **禁止自动执行测试命令覆盖未保存的改动**：先确认文件已保存
- **禁止自动添加/删除依赖**：修改 `pyproject.toml` 的 dependencies 必须先确认

### 安全回退保障
- 每次任务开始前，建议用户确保工作区干净（`git status`）
- 大规模修改前，建议创建临时分支
- 修改后提供回退命令（如 `git checkout -- <file>`）
